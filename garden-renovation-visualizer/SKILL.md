# Garden Renovation Visualizer

既存の庭写真をベースに、改修後の完成予想図（フォトリアル）を生成するスキル。
Codex の built-in `image_gen` ツール（gpt-image-2）の **edit モード** を使う。

## トリガー

- 「この庭の写真から完成予想図を作って」「○○を△△に変えたイメージを生成して」のような指示
- 入力: 庭・外構の写真（添付 or ローカルパス）+ 自然言語の改修要望

## 出力

```
garden-design-output/{YYYY-MM-DD}-{slug}/
├── before.png       # 元写真のコピー
├── after.png        # 完成予想図（最終採用版）
├── after-v2.png …   # イテレーション版（あれば）
└── plan.md          # 改修プランの記録（garden-renovation-siteplan スキルの入力になる）
```

出力先は、ユーザーが指定した場所があればそこ。無ければ**カレントディレクトリ直下の `garden-design-output/`** フォルダを使う。

`{slug}` は案件を表す短い英語スラッグ（例: `backyard-deck`）。

## ワークフロー

### Step 1: 入力の確認

- 写真が無い → 「庭の写真（ファイルパス or 添付）を指定してください」と返して終了
- 写真がローカルパスで指定された場合、**必ず built-in `view_image` ツールで読み込む**（built-in edit は会話コンテキストに見えている画像しか編集できないため）
- 要望が無い・曖昧すぎる場合は、何をどう変えたいか質問してから進む

### Step 2: 現状分析と「保持リスト」の作成

写真を観察し、写っている要素を 2 つに分類する:

- **変更対象**: 要望で変更を指示された部分（例: 芝生 → ウッドデッキ）
- **保持対象（invariants）**: それ以外すべて。最低限、以下は必ず保持リストに入れる:
  - 建物本体（外壁・窓・屋根・玄関）
  - 隣家・ブロック塀・フェンスなど境界構造物（変更指示が無い限り）
  - カメラアングル・パース・画角
  - 太陽光の向き・天候・季節感・空
  - 変更指示の無い植栽・物置・室外機などの設置物

分類に迷う要素（例: 「庭を明るく」と言われたとき既存フェンスを塗り替えてよいか）があれば、**推測せずユーザーに確認する**。

### Step 3: 編集プロンプトの構築

英語で、以下の構造のプロンプトを組み立てる:

```
Use case: garden renovation visualization (edit of an existing photo)
Edit target: the attached photo of an existing garden
Primary request: [変更内容を具体的に。素材・色・配置・サイズ感まで言語化する]
Preserve exactly, do not change: [Step 2 の保持リストを列挙。建物・視点・光・季節は毎回明記]
Change only: [変更する領域を写真内の位置で特定（例: the lawn area in the foreground left）]
Style: photorealistic, seamlessly matching the original photo's lighting, shadows, color grading, and grain
Avoid: altering the building shape or windows, changing the camera angle, adding objects that were not requested
```

要望が複数ある場合も 1 回の edit にまとめてよいが、**4 項目を超える大改修は 2 段階に分ける**（一度に変更点が多いと保持対象が崩れやすい）。

### Step 4: 生成

- built-in `image_gen` ツールの edit で生成する（CLI fallback は使わない。Step 7 の例外のみ）
- 出力サイズは元写真のアスペクト比に最も近いものを選ぶ:
  - 横長 → `1536x1024` / 縦長 → `1024x1536` / 正方形に近い → `1024x1024`
- 複数案を求められたら、同じプロンプトで別々に 1 案 = 1 呼び出しする

### Step 5: 検証

生成画像を表示・確認し、以下のチェックリストで検証する:

- [ ] 建物の形状・窓の位置と数・屋根が元写真と一致しているか
- [ ] カメラアングル・パースが維持されているか
- [ ] 変更指示の無い構造物・植栽が残っているか
- [ ] 要望どおりの変更が反映されているか
- [ ] 光の向き・季節感が元写真と矛盾していないか

崩れがあれば **1 回の再生成につき修正点を 1 つに絞って** イテレーションする。
直前の生成画像を edit 対象にするか元写真からやり直すかは、崩れの程度で判断する（保持対象が崩れた場合は元写真からやり直す方が安全）。

### Step 6: 保存と plan.md の作成

- 出力フォルダを作成し、`before.png`（元写真コピー）と `after.png`（採用版）を保存する
- 生成画像は `~/.codex/generated_images/` に置きっぱなしにせず、必ず出力フォルダへコピーする
- `plan.md` を以下の形式で書く（平面図スキルの入力になるため、位置関係を必ず言語化する）:

```markdown
# 改修プラン: {案件名}

- 日付: {YYYY-MM-DD}
- 元写真: before.png / 完成予想図: after.png

## 要望（原文）
{ユーザーの要望をそのまま}

## 変更内容
- {変更点1: 場所・素材・サイズ感}
- {変更点2: …}

## 保持した要素
- {建物、フェンス、既存植栽 など}

## 敷地の位置関係（写真からの推定）
- {例: 建物は写真奥。手前に芝生、左にブロック塀、右に駐車スペース}
- 実寸の情報: {ユーザーから聞いていれば記載。無ければ「なし（写真からの目視推定のみ）」}
```

- 最後に保存先フルパス・使用したプロンプト・チェックリスト結果を報告する

### Step 7: 厳密な部分保持が必要な場合（例外・要確認）

「建物まわりは 1 ピクセルも変えたくない」など厳密なマスク編集を求められた場合のみ:

- マスク指定・`--input-fidelity high` は CLI fallback（`scripts/image_gen.py` の `edit`）でしか使えず、しかも `gpt-image-1.5` 限定（モデルダウングレード）かつ `OPENAI_API_KEY` が必要であることを説明する
- **ユーザーが明示的に了承した場合のみ** CLI fallback を使う。勝手に切り替えない

## 能力の限界（ユーザーへの説明事項）

完成予想図を渡すとき、初回は以下を一言添える:

- gpt-image-2 の編集は再生成ベースのため、「保持」を指定しても**ピクセル単位の同一性は保証されない**（壁のテクスチャや葉の細部は描き直される。建物の形状・構図レベルでは高精度に保持される）
- 表札・看板など写真内の文字、人の顔は変化しやすい
- 出力解像度はモデル依存（実用上限はおよそ 2K。元写真がそれ以上でも超えられない）
- あくまで「完成イメージの共有」用。施工図・契約資料には使わない

## エラー時の動作

| 状況 | 動作 |
|------|------|
| 写真の指定なし | 写真の提供を求めて終了 |
| 要望が曖昧 | 具体的に何を変えたいか質問する |
| 2 回イテレーションしても保持対象が崩れる | 現状の最良版を提示し、限界と CLI fallback の選択肢を説明する |
| image_gen 失敗 | エラー内容を報告。CLI fallback は提案のみ（勝手に実行しない） |

## 注意

- 自動で次の平面図生成には進まない。「平面図も作る場合は garden-renovation-siteplan スキルで `plan.md` を指定してください」と案内するだけにする
- 元写真は絶対に上書きしない（`before.png` はコピー）
